# 📣 마케팅팀 업무 매뉴얼 (직원교육)

> 제네시스 부업 허브를 **검색에 잘 걸리고, 신뢰가 가고, 클릭하고 싶게** 만드는 팀입니다. 우리는 없는 걸 지어내지 않고, 있는 걸 잘 보이게 합니다.

이 매뉴얼은 마케팅팀에 새로 합류한 직원이 **첫날부터 실무를 시작**할 수 있도록 쓰였습니다. 회사 전반 규칙은 저장소 루트의 `CLAUDE.md`, 발행 절차는 `docs/publishing-guide.md`를 함께 보세요. 우리 역할 정의 원본은 `.claude/agents/marketing-team.md`에 있습니다.

## 1. 우리 팀은 무엇을 하나 (홍보·카드뉴스·SEO)

제네시스는 AI가 운영하는 회사이고, 우리 제품은 **부업 정보 허브 사이트**입니다.

- **사이트 성격:** 한국어, 순수 HTML/CSS/JS(프레임워크·빌드 없음), GitHub Pages `https://parkkweonje.github.io/genesis/` 하위 경로에서 열림.
- **카테고리 4개:** `start`(부업 시작하기) · `online`(온라인·재택) · `ai`(AI·자동화) · `money`(무자본·세금·실무).
- **미션:** 매일 글을 발행·점검하고, **검색 노출(SEO)이 강한 사이트**를 만든다. 콘텐츠 20편을 넘기면 애드센스를 신청한다(지금은 광고 미도입 상태).

마케팅팀이 실제로 만드는 산출물은 세 가지입니다.

| 구분 | 무엇 | 예 |
|------|------|-----|
| **홍보 카피** | 슬로건·헤드라인·소개문·SNS 게시글 | "무자본으로 오늘 시작하는 부업, 제네시스" |
| **카드뉴스** | 인스타/블로그용 5~8장 구성안 | "직장인 부업 5選" 8장 시리즈 |
| **SEO** | title·meta·OG·JSON-LD·sitemap·robots·내부링크 | 새 글의 `<head>` 메타 스니펫 |

우리는 **문구와 구조를 설계**하는 팀입니다. 화면에 실제로 반영(코드·디자인 적용)하는 것은 개발팀·디자인팀과 협업합니다(→ 5장).

## 2. 매일 하는 일 (SEO 점검, 카피, 카드뉴스, SNS)

제네시스는 **매일 글을 발행하고 점검**합니다. 마케팅팀의 하루는 대략 이렇게 흘러갑니다.

1. **오늘 발행 글 확인** — 기획·개발이 올린 새 글(또는 발행 예정 글)을 읽고 무엇을 홍보할지 파악한다.
2. **SEO 점검** — 새 글과 최근 글의 `title`/`description`/OG/JSON-LD/canonical/내부링크가 규칙대로인지 점검한다(→ 3장, `docs/publishing-guide.md`의 체크리스트).
3. **sitemap·robots 확인** — 새 URL이 `sitemap.xml`에 들어갔는지, `robots.txt`가 크롤을 막고 있지 않은지 확인한다.
4. **카피 작성** — 그날 글의 제목·설명·SNS 문구를 여러 안으로 뽑고 추천안을 표시한다.
5. **카드뉴스 구성안** — 주요 글은 카드뉴스 5~8장 구성안으로 만들어 `marketing/` 폴더에 저장한다.
6. **SNS 게시글** — 채널 톤에 맞춘 게시글 + 해시태그를 준비한다.
7. **키워드·트렌드 확인(선택)** — 필요하면 검색·데이터랩 등으로 검색 의도를 확인한다. **확인 안 된 수치는 단정하지 않는다.**
8. **보고** — 산출물과 점검 결과를 마스터에게 간결히 보고한다(무엇을·왜·검증 여부).

> 참고: 콘텐츠 캘린더, 키워드 리스트, 카드뉴스 소재 정리 같은 것은 연결된 커넥터(Google Drive·Notion·Calendar 등)에 자유롭게 정리해도 됩니다(→ 7장).

## 3. SEO 실무 (title/meta/OG/JSON-LD/sitemap/robots/내부링크)

SEO는 우리 팀의 핵심 무기입니다. **메타는 반드시 실제 페이지 내용과 일치**해야 합니다(내용과 다른 메타는 검색·애드센스 모두에서 감점).

### 3.1 검색 의도·키워드 먼저

문구를 쓰기 전에 "이 글을 검색할 사람은 무엇이 궁금한가"를 먼저 정합니다.

- **타깃 키워드 1개 + 보조 키워드 1~2개**를 정한다. (예: 주 = "직장인 부업", 보조 = "퇴근 후", "무자본")
- **검색 의도 유형**을 구분한다: 정보형("~하는 법"), 비교형("~ vs ~"), 거래형("~ 신청"). 우리 글은 대부분 **정보형**이다.
- 키워드는 **제목 앞쪽 → 첫 문단 100자 안 → H2** 순으로 자연스럽게 배치한다. 억지로 반복(키워드 스터핑)하지 않는다.

### 3.2 title / meta description

우리 사이트의 실제 예시(`posts/chatgpt-bueop/`)를 기준으로 합니다.

```html
<title>ChatGPT 부업하는 법 5가지: 초보도 바로 시작</title>
<meta name="description" content="ChatGPT 부업, 어디부터 시작할지 막막하다면 여기부터. 초보도 바로 따라 할 수 있는 현실적인 활용법 5가지를 난이도와 준비물까지 정리했습니다." />
```

- **title:** 타깃 키워드를 앞쪽에, **30~35자 내외**, 페이지마다 **고유**하게.
- **description:** **70~80자**, 키워드 + 클릭 유도 문구. 과장("무조건 월 300")이 아니라 **가치 요약**("난이도와 준비물까지 정리").

### 3.3 Open Graph / Twitter Card

SNS 공유 시 보이는 미리보기입니다. 아래 항목을 title/description과 **일관되게** 채웁니다.

```html
<meta property="og:title" content="…" />
<meta property="og:description" content="…" />
<meta property="og:type" content="article" />
<meta property="og:url" content="https://parkkweonje.github.io/genesis/posts/{slug}/" />
<meta property="og:image" content="/genesis/assets/og-default.png" />
<meta name="twitter:card" content="summary_large_image" />
<meta name="twitter:title" content="…" />
<meta name="twitter:description" content="…" />
<meta name="twitter:image" content="/genesis/assets/og-default.png" />
```

- 현재 대표 이미지는 공통 `/genesis/assets/og-default.png`. 글별 이미지를 만들면 `/genesis/assets/og-{slug}.png`로 저장하고 og:image·twitter:image·JSON-LD image를 함께 교체.
- `canonical`은 `<link rel="canonical" href="https://parkkweonje.github.io/genesis/posts/{slug}/" />`로 정확히.

### 3.4 JSON-LD 구조화 데이터

글 페이지는 `Article` 타입을 씁니다. 필수 필드가 실제 값과 맞는지 확인합니다.

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "제목(=H1과 동일)",
  "description": "메타 설명과 일치",
  "datePublished": "YYYY-MM-DD",
  "dateModified": "YYYY-MM-DD",
  "author": { "@type": "Organization", "name": "제네시스" },
  "publisher": { "@type": "Organization", "name": "제네시스",
    "logo": { "@type": "ImageObject", "url": "https://parkkweonje.github.io/genesis/assets/og-default.png" } },
  "mainEntityOfPage": { "@type": "WebPage", "@id": "https://parkkweonje.github.io/genesis/posts/{slug}/" },
  "image": "https://parkkweonje.github.io/genesis/assets/og-default.png"
}
```

- `headline`은 H1과, `description`은 meta description과 **일치**시킨다.
- 날짜·저자·URL을 실제와 다르게 쓰지 않는다(허위 구조화데이터는 정책 위반).

### 3.5 sitemap.xml / robots.txt

- 새 글을 발행하면 `sitemap.xml`에 `<url>` 블록을 추가한다.

```xml
<url>
  <loc>https://parkkweonje.github.io/genesis/posts/{slug}/</loc>
  <lastmod>YYYY-MM-DD</lastmod>
  <changefreq>monthly</changefreq>
  <priority>0.7</priority>
</url>
```

- `robots.txt`는 현재 전체 허용 + 사이트맵 안내 상태이며, **함부로 바꾸지 않는다**.

```
User-agent: *
Allow: /
Sitemap: https://parkkweonje.github.io/genesis/sitemap.xml
```

### 3.6 내부링크

- 새 글에는 관련 글·카테고리로 향하는 **내부링크 2~3개**를 넣는다.
- 새 글은 **카테고리 목록**(`category/{slug}/`)과 **홈 최신글**(`index.html`의 `#latest`)에도 링크를 추가해 고립되지 않게 한다.
- 앵커 텍스트는 "여기 클릭"이 아니라 **대상 내용을 설명하는 문구**("직장인 부업 체크리스트")로.

> 경로 규칙: 모든 내부 링크·에셋·canonical·og:url·sitemap URL은 반드시 `/genesis/`로 시작하는 루트 상대경로(또는 절대 URL)로 작성합니다.

## 4. 카드뉴스·카피 작성법

### 4.1 카드뉴스 구성 (장별 구성 표)

인스타/블로그용 **5~8장**. 각 장은 **제목 + 핵심 문장 한두 줄 + 비주얼 지시**로 구성합니다. 아래는 8장 표준 골격입니다.

| 장 | 역할 | 제목(예) | 본문 한두 줄 | 이미지 아이디어 |
|----|------|----------|--------------|-----------------|
| 1 | 후킹(표지) | 직장인 부업, 뭐부터? | 퇴근 후 1시간으로 시작하는 5가지 | 큰 타이포 + 브랜드 컬러 배경 |
| 2 | 공감/문제 | "시간이 없어서요" | 하루 30분이면 충분한 것부터 고른다 | 시계·달력 아이콘 |
| 3 | 핵심 1 | ① 블로그·글쓰기 | 무자본, 꾸준함이 자산 | 노트북 일러스트 |
| 4 | 핵심 2 | ② AI 활용 | 초안은 AI, 검토는 나 | 챗 UI 목업 |
| 5 | 핵심 3 | ③ 제휴·콘텐츠 | 링크가 아니라 신뢰를 판다 | 카드형 리스트 |
| 6 | 주의점 | 시작 전 체크 | 세금·정산은 공식 안내 확인 | 체크리스트 |
| 7 | 요약 | 오늘의 3줄 | 고르고 · 30분 · 매일 | 3줄 강조 |
| 8 | CTA | 더 알아보기 | 제네시스 부업 허브에서 | 사이트명 + 카테고리 안내 |

- **5장 버전**은 1(표지) → 3~5(핵심) → 8(CTA)만 남기면 됩니다.
- 마지막 CTA는 **가입 유도·개인정보 수집이 아니라** "사이트에서 더 읽기" 수준으로.

### 4.2 톤

- **친근하지만 과장하지 않음.** "무조건", "보장", "누구나 월 OOO만원" 금지.
- **행동 중심 동사**로: "정리했습니다", "따라 하면 됩니다", "확인하세요".
- **한국어 자연스럽게.** 번역투·불필요한 영어 남발 피함.
- 채널별: 인스타는 짧고 감각적 + 해시태그, 블로그는 조금 더 설명적, 검색(메타)은 담백·정확.

### 4.3 카피는 항상 여러 안 제시

카피는 **3개 내외의 안 + 추천 1안과 이유**로 제출합니다.

- (A) 정보형: "직장인 부업 5가지, 난이도까지 정리"
- (B) 공감형: "퇴근 후 1시간, 뭘 해야 할까"
- (C) 실행형: "오늘 바로 시작하는 부업 체크리스트"
- **추천:** (A) — 검색 의도(정보형)와 맞고 title 키워드가 앞에 옴.

## 5. 업무 흐름 & 협업

### 5.1 회사 업무 흐름 속 우리 위치

제네시스의 흐름은 **접수 → 분류 → 분배 → 검수 → 보고 → 배포**입니다. 조직은 사장 – 마스터(검수·분배) – 기획/개발/디자인/마케팅/CS. 보통 순서는 **기획 → 개발 → 디자인 → 마케팅**입니다. 우리는 대개 **마지막 단계에서 홍보·SEO를 붙입니다.**

### 5.2 개발·디자인과의 경계

| 우리가 하는 것 | 개발/디자인이 하는 것 |
|----------------|------------------------|
| 문구(카피)·메타 텍스트·JSON-LD **내용** 설계 | 그 문구를 **실제 HTML/화면에 반영** |
| 카드뉴스 **구성안**(장별 텍스트·비주얼 지시) | 이미지·레이아웃 **실제 제작**(디자인) |
| sitemap/robots에 넣을 **URL·값 제안** | 파일 반영·경로·빌드 관점 **검토**(개발) |

> **문구는 마케팅이 쓰되, 화면 반영은 개발·디자인과 협업**합니다. 우리가 코드 파일을 직접 고쳐야 할 때도 **요청된 최소 범위**만 손대고, 관련 없는 리팩토링은 하지 않습니다.

### 5.3 산출물은 `marketing/` 폴더에

- 카피·카드뉴스·SNS·키워드 리서치 등 우리 산출물은 저장소의 **`marketing/` 폴더**에 정리해 저장합니다.
- 파일명은 알아보기 쉽게: `marketing/cardnews-jikjangin-bueop.md`, `marketing/copy-chatgpt-bueop.md` 처럼.
- 콘텐츠 캘린더·소재 아카이브는 커넥터(Drive·Notion 등)에 둬도 되지만, **확정 산출물은 저장소에 남깁니다.**

### 5.4 배포

- 배포는 **`git push origin main`으로만** 합니다. Netlify/Vercel CLI 등 다른 경로 금지.
- 사장(사용자) 승인 후 배포하며, 배포·서버 구성을 임의로 바꾸지 않습니다.

## 6. 반드시 지킬 원칙

### 6.1 작업 우선순위

> **정확성 > 검증 > 최소 변경 > 명확성 > 유지보수성**

카피가 아무리 멋져도 **사실과 다르면 폐기**입니다. 먼저 정확하고, 검증하고, 최소로 바꾸고, 명확하게, 나중에 볼 사람이 이해하도록.

### 6.2 거짓·과장 금지 (애드센스 정책 직결)

- **수익 보장·허위 효과 금지.** "이 방법이면 무조건 월 300", "누구나 성공" 같은 표현 금지.
- **확인 안 된 수치를 단정하지 않는다.** 통계·수익·순위 수치는 출처를 붙이거나, 없으면 쓰지 않는다.
- **세금·법률·정책 수치**는 "국세청 등 공식 안내 확인" 문구를 함께 넣는다.
- **개인정보·민감정보 수집을 유도하지 않는다.** ("이메일 남기면 비법 전수" 식 금지)
- **메타는 실제 내용과 일치.** 낚시성 제목/설명(내용에 없는 것 약속) 금지.

이 원칙들은 단순 매너가 아니라 **애드센스 승인·유지 조건**입니다. 20편 후 애드센스를 신청할 때 위반 문구 하나가 승인을 막을 수 있습니다.

## 7. 접근 정책

### 🚫 절대 금지 — 사용자 로컬 PC의 '개인' 폴더

- 사용자 로컬 PC **바탕화면(Desktop)의 '개인' 폴더는 사적 공간**입니다. **어떤 경우에도 접근하지 않습니다.**
- 소재를 찾는다는 이유로도 개인 폴더를 열거나 읽지 않습니다. 필요한 자료는 사장에게 요청합니다.

### ✅ 자유 사용 — 연결된 커넥터

- **Google Drive · Notion · Calendar** 등 연결된 도구는 업무에 **자유롭게 활용**합니다.
- 활용 예: **콘텐츠 캘린더** 관리(Calendar/Notion), **카드뉴스·키워드 소재 정리**(Drive/Notion), 발행 일정 공유.
- 단, 커넥터에 올리는 내용에도 6장 원칙(거짓·과장 금지, 개인정보 수집 유도 금지)이 동일하게 적용됩니다.

## 8. 발행·홍보 체크리스트

새 글을 홍보할 때 아래를 확인합니다. (발행 절차 자체는 `docs/publishing-guide.md`)

- [ ] **title** 고유 · 키워드 앞쪽 · 30~35자 내외
- [ ] **meta description** 70~80자 · 키워드 + 클릭 유도 · 과장 없음
- [ ] **OG/Twitter** og:title/description/type=article/url/image, twitter:card=summary_large_image, title과 일관
- [ ] **canonical** `https://parkkweonje.github.io/genesis/posts/{slug}/` 정확
- [ ] **JSON-LD** Article, headline=H1, description=meta, 날짜·author·mainEntityOfPage 정확
- [ ] **내부링크** 관련 글/카테고리 2~3개, 앵커 텍스트 설명형
- [ ] **카테고리·홈 노출** 카테고리 목록 + 홈 `#latest`에 링크 추가
- [ ] **sitemap.xml** 새 URL `<url>` 블록 추가
- [ ] **robots.txt** 크롤 차단 없음 (임의 변경 금지)
- [ ] **경로** 모든 링크·에셋이 `/genesis/`로 시작
- [ ] **카피** 3안 + 추천 1안, 과장·수익 보장 없음
- [ ] **카드뉴스/SNS** 구성안 + 해시태그 준비, `marketing/`에 저장
- [ ] **원칙 점검** 거짓·과장·개인정보 유도 없음 (애드센스 정책 통과)
- [ ] **배포** 승인 후 `git push origin main`

## 9. 자주 하는 실수 & 금지사항

- ❌ **낚시성 메타** — 제목/설명에서 약속한 걸 본문이 안 지킴. (메타는 내용과 일치)
- ❌ **수익 보장·과장** — "무조건", "보장", "누구나 월 OOO". 절대 금지.
- ❌ **출처 없는 수치 단정** — 확인 안 된 통계·순위를 사실처럼 씀.
- ❌ **개인정보 수집 유도** — "이메일 남기면 비법" 식 카피.
- ❌ **키워드 스터핑** — 같은 키워드를 부자연스럽게 반복. 가독성·SEO 모두 손해.
- ❌ **경로 실수** — 내부 링크·og:url·canonical을 `/genesis/` 없이 작성. GitHub Pages 하위경로에서 깨짐.
- ❌ **카피 1개만 제출** — 항상 3안 + 추천안.
- ❌ **화면 직접 대폭 수정** — 문구 반영은 개발·디자인과 협업. 코드 손댈 땐 최소 범위만.
- ❌ **robots/sitemap 임의 변경** — 규칙대로 URL 추가만. 크롤 차단·구조 변경 금지.
- ❌ **개인 폴더 접근** — 사용자 Desktop '개인' 폴더 열람 금지(7장).
- ❌ **다른 경로로 배포** — 배포는 오직 `git push origin main`.
